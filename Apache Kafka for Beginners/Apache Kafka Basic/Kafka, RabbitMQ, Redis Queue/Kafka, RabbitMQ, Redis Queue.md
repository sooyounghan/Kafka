### Kafka, RabbitMQ, Redis Queue의 차이점
-----
1. 메세징 플랫폼
2. 메시지 브로커 : 이벤트 브로커로 역할을 할 수 없음 (Redis Queue, RabbitMQ)
   - 대규모 메세징 기반 미들웨어 아키텍쳐에서 사용되어 왔음 (미들웨어(Middleware) : 서비스하는 애플리케이션들을 보다 효율적으로 아키텍쳐들을 연결하여 동작하는 소프트웨어로, 메세지 플랫폼 / 인증 플랫폼 / 데이터베이스 등) 
   - 메세지 브로커에 있는 큐에 데이터를 보내고 받는 프로듀서와 컨슈머를 통해 메세지를 통신하고 네트워크를 맺는 용도로 사용
   - 메세지를 받아 적절히 처리하고 나면, 즉시 또는 짧은 시간 내 삭제되는 구조

3. 이벤트 브로커 : 메세지 브로커로 역할을 할 수 있음 (Kafka, AWS의 키네시스)
   - 이벤트 또는 메세지라고 불리는 레코드를 하나만 보관하고, 인덱스를 통해 개별 액세스를 관리 
   - 업무 상 필요한 시간 동안 이벤트를 보존할 수 있음
   - 메세지 브로커와 데이터를 처리해도 삭제하지 않음
      + 서비스에서 나오는 이벤트를 DB에 저장하듯이 이벤트 브로커 큐에 저장
      + 딱 한 번 일어난 이벤트 데이터를 브로커에 저장함으로, 단일 진실 공급원으로 사용 가능
      + 장애가 발생했을 때, 장애가 일어난 지점부터 재처리 가능
      + 많은 양의 실시간 스트림 데이터를 효과적으로 처리 가능

   - 이벤트 브로커로 클러스터를 구축하면, 이벤트 기반 MSA로 발전하는데 중요한 역할을 하며, 메세지 브로커로도 사용할 수 있음
